## High

### [H1] `listPublicTemplates` 无数据库分页，全表加载后内存排序

- **文件：** `src/lib/public-templates.ts:69-87`
- **证据：**
  ```ts
  const rows = await prisma.template.findMany({
    where,
    select: {
      id: true, name: true, description: true, category: true,
      ratingCount: true, ratingSum: true, updatedAt: true,
      steps: { orderBy: { order: "asc" }, select: { role: true } },
      _count: { select: { forks: true } },
    },
  });
  // …全量 map → sortTemplates(mapped) → arr.slice(start, start + pageSize)
  ```
- **影响：** REST `/api/templates/public` 与 MCP `list_public_templates` 均共享此实现。当前库约数十条尚可承受，但 seed 脚本每批次新增 6–10 条，每次请求仍把全部 public 模板（含 steps 关联）捞到 Node 端排序；条目上量（>500）后响应时间会线性劣化，且对 Postgres 造成不必要的 I/O。`pageSize` 上限 100 无法约束行数。
- **建议：** 推荐排序可离线预计算 `recommendedScore` 并入列 + DB-level `orderBy` + `skip/take`；`popular/top_rated/latest` 直接用 `orderBy + take + skip`。`recommended` 若需复合公式，至少先分页（按 `updatedAt DESC`）再排序，或后台 cron 写入 `Template.recommendedScore` 字段。

### [M2] `TemplateTestRunner` 成本累加丢失 Decimal 精度

- **文件：** `src/lib/template/test-runner.ts:200-207`
- **证据：**
  ```ts
  const sellPrice = callLog?.sellPrice ? callLog.sellPrice.toString() : "0";
  if (stepTotal) totalTokens += stepTotal;
  if (callLog?.sellPrice) {
    totalCostUsd += Number(callLog.sellPrice);
  }
  // …
  const totalCost = totalCostUsd.toFixed(8);
  ```
- **影响：** `sellPrice` 在数据库里是 `Decimal`。先 `Number()` 再 `.toFixed(8)` 的做法，`toFixed(8)` 只是格式化，累加阶段的浮点舍入已发生。单次测试成本极小，8 位小数在单位美分级尚可承受，但同一 traceId 的多步骤 execute 模式下（规格最多 20 条 run × 多步 action），长期累积会与 CallLog 实际总和产生可见偏差，影响用户在 UI 看到的"测试成本"显示。
- **建议：** 改用 `Prisma.Decimal` 累加：`totalCostUsd = Prisma.Decimal.sum(totalCostUsd, new Prisma.Decimal(callLog.sellPrice))`，结束后 `.toFixed(8)`；或直接保存最后一条 run 的实际 sellPrice 汇总值。

### [M3] `waitForCallLog` 顺序轮询使多步模板额外等待最多 3s × N

- **文件：** `src/lib/template/test-runner.ts:247-262`
- **证据：**
  ```ts
  const CALL_LOG_POLL_INTERVAL_MS = 100;
  const CALL_LOG_POLL_MAX_ATTEMPTS = 30;
  // …
  async function waitForCallLog(traceId: string) {
    for (let attempt = 0; attempt < CALL_LOG_POLL_MAX_ATTEMPTS; attempt++) {
      const log = await prisma.callLog.findUnique({ where: { traceId }, /* … */ });
      if (log) return log;
      await new Promise((resolve) => setTimeout(resolve, CALL_LOG_POLL_INTERVAL_MS));
    }
    return null;
  }
  ```
- **影响：** 每步 action 结束后都会串行轮询 CallLog 最多 3 秒。TEMPLATE-TESTING 支持多步模板（如 2-step / 3-step），worst-case 一次 `execute` 测试在"CallLog 写入比 response 慢"的边界场景下累计等待 6–9 秒，被前端感知为卡顿，且占用 API route 的连接时间。规模化后还会放大到 MCP `run_template` 调用路径。
- **建议：** 把 CallLog 改为由 `runActionNonStream` 返回时直接附带（同事务或同请求上下文），或把 timeout 降至 1s；再不济，至少在 `runActionNonStream` 内同步等待 CallLog 写完后才 return。
