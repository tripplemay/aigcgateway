# P1.6 模型同步引擎重构 — 方案演进记录

## 背景

P1.5 的同步引擎依赖硬编码数据：每家服务商的模型价格、上下文窗口等信息都预填在 `pricingOverrides` 或 `staticModels` 中。火山引擎仅有 4-7 个模型（实际提供 30+），其他服务商的价格也需要人工维护。

**核心问题：** 同步引擎无法主动发现服务商的完整模型列表和价格。

## 第一版方案：两层同步 + AI 读文档（初始设计）

### 架构

```
第 1 层：/models API（自动，免费）
  ↓ 拿到模型列表（可能不完整、缺价格）
第 2 层：AI 读服务商文档（自动，低成本）
  ↓ fetch 文档页 HTML → AI 提取模型+价格 → 补全第 1 层缺失数据
```

### 实现

- `doc-enricher.ts`：AI 文档提取层
  - 用 Node.js `fetch()` 获取服务商文档页面 HTML
  - 清洗 HTML（去 script/style/nav），截断到合理长度
  - 调用内部 DeepSeek AI（绕过 API Gateway 鉴权/计费）解析定价数据
  - 与第 1 层数据合并（只补不覆盖）
- `callInternalAI()`：直接使用数据库中 DeepSeek Provider 的凭证发请求，不写 CallLog、不扣费
- 7 家适配器全部移除 `pricingOverrides` 依赖
- `pricingOverrides` 从 seed 中清空，不预填
- 每家 Provider 的 `docUrls` 指向其官方定价页

### 7 家服务商策略（初始）

| 服务商 | 第 1 层 | 第 2 层 docUrls |
|--------|---------|----------------|
| OpenAI | /models | `platform.openai.com/docs/pricing` |
| Anthropic | /models | `docs.anthropic.com/en/docs/about-claude/models` |
| DeepSeek | /models | `api-docs.deepseek.com/quick_start/pricing` |
| 智谱 | /models | `open.bigmodel.cn/pricing` |
| 火山引擎 | 无 API | `volcengine.com/docs/82379/1399008` |
| 硅基流动 | /models | `siliconflow.cn/pricing` |
| OpenRouter | /models（含价格）| 不需要 |

### 测试结果

本地验证（有 DeepSeek key）：

- **DeepSeek**: AI 补充 2 条价格 ✅
- **其余 5 家**: AI 提取结果为 0

### 问题发现

`fetch()` 拿到的是原始 HTML，不执行 JavaScript：

- **OpenAI**: 返回 403（反爬保护）
- **Anthropic / 智谱 / 火山引擎**: 返回 SPA 空壳（React/Vue 动态渲染，定价数据在 JS Bundle 中）
- **硅基流动**: 能 fetch 到内容（754KB SSR），但清洗后仍有 10KB 杂乱文本，AI 120 秒超时

**结论：7 家中只有 DeepSeek 的文档页能被 `fetch()` 有效获取。**

## 第二版方案：承认现实，分类处理（中间调整）

### 设计原则调整

> "能自动化的自动化，不能的就承认它需要运营手动维护，不伪装。"

### 分类策略

| 服务商 | 路径 | 原因 |
|--------|------|------|
| DeepSeek | AI 读文档 | 文档页可 fetch |
| 硅基流动 | AI 读文档 | 文档页可 fetch（但后来发现超时） |
| OpenAI / Anthropic / 智谱 | `pricingOverrides` 手动维护 | 文档页 403 / SPA |
| 火山引擎 | `staticModels` 手动维护 | 无 API，文档页 SPA |
| OpenRouter | 纯 API | 含完整价格 |

### 中间曾尝试过的 `text:` 内嵌方案（已废弃）

为解决 SPA 页面问题，曾在 `docUrls` 中使用 `text:` 前缀内嵌纯文本：

```typescript
docUrls: ["text:火山引擎方舟模型列表...model_id: doubao-1.5-pro-256k..."]
```

doc-enricher 检测到 `text:` 前缀时跳过 fetch，直接将文本交给 AI 解析。

**这个方案被否决。** 原因：本质上是把硬编码数据从 `pricingOverrides` 搬到了 `docUrls`，换了个字段名，还多了一次 AI 调用去解析人工写的文本。比直接用 `pricingOverrides` 更差。

### 硅基流动从 AI 路径移除

硅基流动虽然能 fetch 到页面内容，但 754KB SPA 清洗后 10KB 交给 AI，DeepSeek 120 秒内无法完成 95 个模型的价格提取。多次生产验证均超时 abort。

**最终判定：** 硅基流动也归入"手动维护"类别，docUrls 设为 null。

### 生产验证状态

- 火山引擎 14 个 staticModels 生效 ✅
- DeepSeek AI 补充 2 条价格 ✅
- 硅基流动 95 个模型价格为 0（预期行为）
- OpenAI / Anthropic / 智谱由 pricingOverrides 提供价格 ✅

**此时自动价格发现覆盖率：2/7（DeepSeek + OpenRouter）。**

## 第三版方案：Jina Reader 突破内容获取瓶颈（最终方案）

### 关键发现

用户指出：这些服务商文档都是通过 CLI 用 AI 读取下来的，也是在 CLI 环境，为什么我们遇到了障碍？

**根因不是"反爬"，是我们用了错误的工具。** `fetch()` 是纯 HTTP 客户端，不执行 JS。而 CLI 工具（如 Claude Code 的 WebFetch）背后有渲染服务。

### 解决方案：Jina Reader

[Jina Reader](https://r.jina.ai/) 是免费的网页渲染 + 内容提取服务，将任意 URL 转为干净的 Markdown 文本：

```
fetch("https://r.jina.ai/" + url)  // 返回渲染后的 markdown
```

### 验证结果

| 服务商 | 之前 fetch | Jina Reader | 有效内容 |
|--------|-----------|-------------|---------|
| OpenAI | 403 | ✅ 200 | 定价数据完整 |
| Anthropic | SPA 空壳 | ✅ 渲染完成 | 模型信息完整 |
| 智谱 | SPA 空壳 | ✅ 渲染完成 | 定价表完整 |
| 硅基流动 | 754KB SPA | ✅ 24KB markdown | 定价数据完整 |
| 火山引擎 | SPA 空壳 | ✅ 29KB markdown | 模型列表完整 |

### 代码变更

`fetchDocPage()` 改为通过 Jina Reader 获取：

```typescript
// 之前：直接 fetch（无法处理 SPA 和反爬）
const response = await fetch(url, { headers });

// 之后：通过 Jina Reader 渲染后获取 markdown
const response = await fetch(`https://r.jina.ai/${url}`, {
  headers: { Accept: "text/plain" },
});
```

不再需要 HTML 清洗逻辑（Jina 直接返回 markdown）。

### Seed 变更

所有 7 家（除 OpenRouter）都配 docUrls，移除全部 pricingOverrides 和 staticModels：

```typescript
// OpenAI — 之前需要 pricingOverrides 手动维护 8 个模型价格
// 现在 AI 自动从 pricing 页提取
docUrls: ["https://platform.openai.com/docs/pricing"],
pricingOverrides: null,

// 火山引擎 — 之前需要 staticModels 手动维护 14 个模型
// 现在 AI 自动从文档提取
docUrls: ["https://www.volcengine.com/docs/82379/1399008"],
staticModels: null,
```

### 最终 7 家策略

| 服务商 | 第 1 层 (API) | 第 2 层 (Jina + AI) | 手动维护 |
|--------|-------------|---------------------|---------|
| OpenAI | /models 白名单 | ✅ AI 提取价格+上下文 | 无 |
| Anthropic | /models（含 context） | ✅ AI 补充价格 | 无 |
| DeepSeek | /models | ✅ AI 补充价格 | 无 |
| 智谱 | /models | ✅ AI 补充价格（22 个模型） | 无 |
| 火山引擎 | 无 API | ✅ AI 提取全部模型+价格 | 无 |
| 硅基流动 | /models | ✅ AI 补充价格 | 无 |
| OpenRouter | /models（含完整价格） | 不需要 | 无 |

anthropic:   3 models  (API: 0, AI: +3)  — 不稳定，有时提取 0

### 已知局限

- Anthropic 和火山引擎的 AI 提取不稳定（有时返回 0），可能与 Jina 返回内容的截断位置有关
- AI 提取结果依赖 DeepSeek 的 JSON 解析能力，偶尔返回非法格式
- Jina Reader 是第三方免费服务，无 SLA 保证
- `pricingOverrides` 和 `staticModels` 字段保留作为运营兜底入口——当 AI 提取不准确时运营可手动修正

## 核心教训

1. **内容获取是瓶颈，不是 AI 解析。** AI 提取定价数据的能力从第一天就够用了，卡在拿不到页面内容。
2. **不要为了架构美感伪装数据来源。** text: 内嵌方案表面上走 AI 管道，实际是硬编码换皮。被否决是正确的。
3. **选对工具比优化算法重要。** 从 `fetch()` 换到 Jina Reader，一行代码的改动解决了所有 SPA/反爬问题。
4. **渐进式验证比一次到位靠谱。** 每个阶段都在生产验证，及时发现问题调整方向，避免了大规模返工。
