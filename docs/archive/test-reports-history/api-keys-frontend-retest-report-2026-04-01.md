# API Keys 前端问题回归报告

## 测试目标

- 针对上一轮本地报告中的 3 个问题执行定向回归：
  - 搜索框清空后列表不恢复
  - ACTIVE 行 `edit` 按钮无行为
  - 列表复制按钮复制掩码值

## 测试环境

- 环境：本地 Codex 测试环境
- 端口：`3099`
- 页面：`http://127.0.0.1:3099/keys`
- 浏览器：Chrome MCP

## 回归结果

### FAIL-001 搜索框清空后列表仍不恢复

- 复现步骤：
  1. 在 `/keys` 搜索框输入 `NoSuchKey`
  2. 页面显示 `No keys found`
  3. 清空搜索框
  4. 等待页面重新渲染
- 实际结果：
  - 搜索框为空后，页面仍停留在 `No keys found`
  - 列表没有自动恢复
- 预期结果：
  - 清空搜索框后应立即恢复完整列表
- 结论：
  - 问题仍未修复

### PASS-001 ACTIVE 行 `edit` 按钮已改为禁用态

- 实际结果：
  - ACTIVE 行 `edit` 按钮存在，但为 disabled
  - 同时带有 `Coming soon` 描述
- 预期结果：
  - 若详情页未在本期范围，应避免“可点击但无行为”的误导
- 结论：
  - 上一轮“无行为按钮”问题已修复为可接受的占位态

### PASS-002 列表复制按钮已从行内移除

- 实际结果：
  - ACTIVE 行现在仅显示掩码值 `pk_xxx...****`
  - 行内不再出现 `content_copy` 复制按钮
- 预期结果：
  - 在后端列表接口只返回掩码值的前提下，不应保留误导性的复制行为
- 结论：
  - 上一轮“复制掩码值”的问题已修复

## 风险项

### RISK-001 本地启动脚本仍存在产物服务不稳定问题

- 使用 `codex-restart.sh` 后，`node .next/standalone/server.js` 对部分 `_next/static/chunks` 返回 `404`
- 为完成本轮回归，改用了 `next start` 方式启动本地服务
- 此问题属于本地测试环境/启动方式风险，不直接等同于 API Keys 页面缺陷

## 最终结论

本轮定向回归结论：**部分通过**。

- 已修复：
  - `edit` 按钮无行为
  - 列表复制掩码值
- 仍未修复：
  - 搜索框清空后列表不恢复
