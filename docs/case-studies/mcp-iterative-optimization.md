# 用 AI 审计 AI：构建 Agent 友好系统的工程实践

> 如何让 AI Agent 自己告诉你，你的系统哪里不够好？
>
> 本文记录了 AIGC Gateway MCP 服务从"能用"到"好用"的六轮迭代。
> 核心方法：让 AI 扮演用户，自主探索系统，产出人无法预见的发现。

---

## 一、问题：什么是"AI 友好"？

当你的系统暴露了一组 API（或 MCP Tools）给 AI Agent 调用时，一个根本问题出现了：

**AI Agent 能否在没有人类辅助的情况下，独立完成端到端的业务任务？**

这不是传统意义上的"API 设计好不好"，而是一个更高的标准：

| 传统 API 设计 | AI 友好设计 |
|--------------|-----------|
| 文档清晰，人类能看懂 | Tool 描述精确，AI 能理解意图 |
| 有错误码，人类能查手册 | 错误信息自解释，AI 能自主排错 |
| 返回 JSON，人类能解析 | 返回结构化数据，AI 能直接推理 |
| 功能完整，人类能手动操作 | 读写闭环，AI 能自主完成全流程 |
| 有测试，人类跑用例 | AI 能自主测试自主发现问题 |

**核心洞察：你无法通过人类视角完整评估 AI 友好性，因为人类太了解自己的系统了。** 你需要让一个"零知识的 AI"来体验你的系统，它的困惑和失败就是你需要修复的地方。

---

## 二、方法：让 AI 当审计员

### 2.1 核心理念

传统测试：人类编写测试用例 → 脚本执行 → 检查结果
AI 驱动测试：**给 AI 一个角色和目标 → AI 自主探索 → AI 产出发现 → 人类决策**

关键区别：AI 不是在执行预定义的检查清单，而是以"第一次接触这个系统的陌生人"身份自主探索。这能发现人类因为太熟悉系统而忽视的盲区。

### 2.2 六个审计角色

我们设计了六个互补的 AI 角色，从不同维度审计系统的 AI 友好性。每个角色通过一段提示词激活，提示词固定不变以确保跨轮次可比。所有提示词存放在 `docs/case-studies/prompt/`。

#### 角色 1：DX 逆向工程师 — 发现"不可理解"

```
你是一位极其挑剔的资深架构师和 DX 审查员。
你刚获得了一个未知系统的 MCP 接入权限。
你没有任何文档或源代码。
```

**任务：** 全量盘点 Tool → 逆向推演系统全貌 → 吐槽设计质量

**为什么有效：** AI 从零理解系统时的困惑，就是真实用户（包括 AI Agent）的困惑。如果 DX 审查员无法从 Tool 描述中推断出 capabilities 字段的含义，那接入你系统的 AI Agent 也不行。

**我们发现了什么：**
- `capabilities` 字段大部分为空对象 — AI 无法判断模型能力
- `price` 是人类可读 string（"$0.336 in / $0.504 out per 1M tokens"）— AI 无法程序化比价
- chat 支持 function calling 但没有模型声明支持 — AI 只能靠试错

#### 角色 2：混沌工程师 — 发现"不安全"

```
你是一位顶级的网络安全与混沌工程师。
你的目标是：尽最大努力去弄坏或迷惑这个系统。
```

**任务：** 资产速览 → Fuzzing（不存在的模型名、极端参数、注入攻击）→ 逐字审查错误信息

**为什么有效：** AI 会尝试人类觉得"没必要测"的场景（JNDI 注入、原型链污染、空字节），而且会**逐字审查**错误信息中的每一个字符。人类扫一眼觉得"错误信息正常"，AI 会发现里面藏着的 QQ 群号。

**我们发现了什么：**
- 无效图片尺寸的错误信息透传了上游供应商的 QQ 群号 `836739524`
- volcengine 错误透传了上游 Request ID 和端点名
- 空 content 未校验，白白消耗了 2510 tokens

**关键提示词技巧：** 在提示词中写明 "如果有上游供应商痕迹，请立刻标红警告" — 这条指令让 AI 把注意力聚焦在人类最容易忽略的安全细节上。

#### 角色 3：FinOps 审计员 — 发现"不精确"

```
你是一位极其严苛的 FinOps 审计员。
你的任务是审查计费精度和数据一致性。
```

**任务：** 基线快照 → 两次定量消耗（极小 + 较大）→ 三方闭环对账（日志 vs 聚合 vs 余额）

**为什么有效：** AI 会做人类懒得做的数学验证 — 手动计算理论费用（5 tokens × $0.18 / 1M = $0.0000009）并与系统记录逐位对比。

**我们发现了什么：**
- 4 位小数精度下，微额调用显示 $0.0000（看似免费但实际扣了费）
- 交易流水没有 trace_id，无法将扣费关联到具体调用
- 审计窗口内出现"幽灵交易"（实际是并行审计员的调用）

**关键提示词技巧：** 要求 "精确到小数点后 4 位" — 定量约束让 AI 产出可验证的结论，而非模糊的"感觉还行"。

#### 角色 4：工作流架构师 — 发现"不完整"

```
你是一位精通低代码与提示词工程的架构师。
你需要评估 Prompt 资产管理和编排的 DX。
```

**任务：** 探查编排工具 → 全生命周期演练（创建 → 版本迭代 → 执行 → 编排 → 清理）→ 评估读写断层

**为什么有效：** AI 会真正走完完整的业务流程，而不是只测单个 API。人类测试倾向于"测 create 通过了就测下一个"，AI 会发现只有 list/get/run 没有 create/update/delete 的**读写断层**。

**我们发现了什么：**
- Action/Template 只能在 Web 控制台创建，MCP 侧无 CRUD — 这是最关键的 AI 不友好
- 无法回滚到旧版本，只能创建新版本
- run_template 只返回最终 output，看不到中间步骤

#### 角色 5：新手通关测试 — 发现"不直觉"

```
模拟零背景新手，4 阶段递进：
阶段一：能力探索 — "我完全不知道这个平台能干什么"
阶段二：代码生成 — "帮我写个脚本调用 DeepSeek"
阶段三：故障排查 — "脚本报错了，帮我查日志"
阶段四：成本感知 — "折腾这一通花了多少钱"
```

**为什么有效：** 这不是测 API，是测 **AI Agent 能否通过你的系统自主完成用户任务**。如果 Agent 在"写脚本调用模型"这一步卡住了，说明 list_models 返回的信息不够结构化，AI 推断不出请求格式。

#### 角色 6：对抗性沙盒 — 发现"不健壮"

在新手测试基础上增加对抗：故意禁用 API Key、耗尽余额，观察 Agent 能否自主排错。

**为什么有效：** 如果 Agent 看到 401 错误只会说"网络请求失败"而不是"你的 Key 被禁用了"，说明你的错误信息不够 Agent 友好。

### 2.3 提示词设计六原则

| 原则 | 说明 | 示例 |
|------|------|------|
| **角色先行** | 定义专业身份，决定审查视角 | "你是一位极其挑剔的资深架构师" |
| **零上下文** | 明确没有文档，只靠 Tool 探索 | "你没有任何外部文档或源代码" |
| **结构化步骤** | 编号步骤约束输出，便于对比 | "严格按照以下 3 个步骤执行" |
| **对抗性目标** | 给破坏者的目标是"弄坏"而非"验证" | "尽最大努力去弄坏这个系统" |
| **红线标注** | 告知 AI 哪些发现需要高亮 | "上游供应商痕迹，立刻标红警告" |
| **定量约束** | 要求精确数字，产出可验证结论 | "精确到小数点后 4 位" |

### 2.4 多角色交叉：一次审计，多类发现

同一个系统，同一轮迭代，不同角色看到完全不同的问题：

```
DX 审查员：capabilities 字段为空      → 发现"不可理解"
混沌工程师：错误信息泄露 QQ 群号      → 发现"不安全"
FinOps 审计员：精度截断显示 $0.0000   → 发现"不精确"
工作流架构师：无法回滚版本             → 发现"不完整"
新手测试：Agent 推断不出 SDK 用法      → 发现"不直觉"
```

**单一角色永远只能发现一类问题。多角色交叉验证是这套方法的核心价值。**

## 三、迭代：每轮修复，每轮复测

### 3.1 闭环流程

```
外部 AI 审计（独立会话，零知识）
    │
    ▼ 产出审计报告
内部 Harness（分析 → 批次规划 → Generator 实现 → Evaluator 验收）
    │
    ▼ 部署到生产
外部 AI 审计（同一提示词，重新审计）
    │
    ▼ 对比上轮结果，发现新问题
    ...循环
```

**关键：审计是在工程流程外部执行的。** AI 审计员不参与 Harness 的 building/verifying 阶段，它只在修复完成并部署后，以零知识用户身份重新体验系统。提示词不变，结果的变化就是改进的证据。

### 3.2 六轮迭代轨迹

#### 第 1-2 轮：从"能用"到"完整"

**触发：** DX 审查员发现 Action/Template 只能在 Web 控制台操作，MCP 侧只有 7 个只读/调用 Tool。

**修复：** 新增 13 个 CRUD Tool（从 7 到 20），补全 chat 的 function calling 参数。

**AI 友好性提升：** Agent 从"只能调用模型"进化到"能自主管理 Prompt 模板全生命周期"。

#### 第 3 轮：从"完整"到"可理解"

**触发：** DX 审查员指出 capabilities 不完整、contextWindow 为 null、错误缺 error_code。

**修复：** 补全模型元数据、新增 dry_run 预览、统一错误格式为 `[error_code] message`。

**AI 友好性提升：** Agent 能通过 capabilities 判断模型能力（而非试错），能通过 error_code 自主排错（而非猜测），能零成本预览 prompt 渲染结果。

**教训：** 这个批次经历了 5 轮修复。error_code 的传播路径有多个 catch 分支，Generator 每轮只修了部分。**对照已有的好模式（generate-image.ts）统一修复**才最终通过。

#### 第 4 轮：从"可理解"到"可审计"

**触发：** FinOps 审计员发现交易无法追溯（缺 trace_id）、精度不足（4 位小数截断）。工作流架构师发现无法回滚版本、看不到中间步骤。

**修复：** 交易增加 trace_id、计费精度升到 8 位、新增 activate_version 和 run_template 步骤明细。

**AI 友好性提升：** Agent 能追溯每笔扣费到具体调用，能回滚 prompt 版本做 A/B 测试，能看到多步编排的每一步执行细节。

#### 第 5 轮：从"可审计"到"安全"

**触发：** 混沌工程师复测发现 generate_image 脱敏仍不完整，FinOps 分析发现理论上的零扣费风险。

**修复：** 全路径脱敏、最低扣费保护（MIN_CHARGE）、空 content 校验。

**安全教训：** generate_image 的上游错误脱敏经历了**三轮**才完全修复（chat 路径 → routing catch → adapter catch + REST API）。**安全修复必须全路径排查，不能只修报告提到的那个。**

#### 第 6 轮：前端全面审查（进行中）

Codex 以 14 个维度审查 20 个控制台页面，产出统一修复清单。

## 四、发现：AI 友好的五个维度

六轮迭代下来，我们归纳出 AI 友好系统的五个核心维度：

### 4.1 可发现（Discoverable）

AI Agent 能否仅通过 Tool 描述和返回值理解系统全貌？

| 不友好 | 友好 |
|--------|------|
| capabilities 为空对象 | 明确标注 `function_calling: true` |
| contextWindow 为 null | 填写具体数值 `128000` |
| price 是 string | 结构化的 `{inputPer1M: 0.18}` |
| 20 个 Tool 但 SERVER_INSTRUCTIONS 只写了 7 个 | 所有 Tool 的使用场景和约束都有说明 |

### 4.2 可操作（Actionable）

AI Agent 能否独立完成端到端的业务流程？

| 不友好 | 友好 |
|--------|------|
| 只能 list/get/run，创建要去 Web 控制台 | 完整 CRUD + Run |
| 只能执行活跃版本 | 支持指定 version_id |
| 不可逆（无法回滚版本） | activate_version 随时切换 |
| 编排结果是黑盒 | 返回每步的 input/output/usage/latency |

### 4.3 可排错（Debuggable）

AI Agent 遇到错误时能否自主诊断？

| 不友好 | 友好 |
|--------|------|
| "Provider temporarily unavailable" | `[channel_unavailable] No active channel for model "xxx". Try another model.` |
| 只有 message，没有 error_code | `[model_not_found]` — Agent 能程序化判断错误类型 |
| 实际调用才能看渲染结果 | dry_run=true 零成本预览 |
| 扣费记录无法追溯 | 每笔交易带 trace_id |

### 4.4 安全透明（Secure & Transparent）

系统是否在保护内部架构的同时给出足够信息？

| 不友好 | 友好 |
|--------|------|
| 错误信息泄露上游 QQ 群号 | sanitizeErrorMessage 过滤所有上游痕迹 |
| 伪造 ID 能访问其他项目数据 | 严格的 projectId 隔离 |
| 空 content 白白消耗 2510 tokens | 网关层前置校验拦截 |
| 微额调用被精度截断为 $0.0000 | MIN_CHARGE 保护 + 8 位精度 |

### 4.5 可预测（Predictable）

系统行为是否一致、无歧义？

| 不友好 | 友好 |
|--------|------|
| Schema 定义 top_p min=0 但运行时拒绝 0 | Schema 与运行时一致 |
| 不可用模型出现在 list_models 中 | 只展示可用模型（或标注状态） |
| 文本错误用 `[code] msg`，图片错误用 JSON | 统一格式 |

## 五、工程实践

### 5.1 两个独立体系

本项目的工程实践由两个**完全独立**的体系组成，各自有不同的目标和身份：

```
┌─────────────────────────────────┐     ┌─────────────────────────────────┐
│   AI 审计体系（外部视角）          │     │   Harness 工程体系（内部执行）     │
│                                 │     │                                 │
│   6 个 AI 审计角色               │     │   Planner → Generator → Evaluator │
│   独立会话、零知识、无源码访问     │ ──→ │   状态机驱动、批次化实现            │
│   产出：审计报告 + 问题清单       │     │   产出：代码修复 + 验收报告         │
│                                 │     │                                 │
│   身份：外部用户 / 审计员         │     │   身份：内部工程团队               │
└─────────────────────────────────┘     └─────────────────────────────────┘
        ↑                                           │
        └───────── 修复后用同一提示词复测 ──────────────┘
```

**为什么要分开？**

AI 审计员必须独立于工程流程之外，原因是：

1. **零知识保证** — 审计员不看源码、不读 CLAUDE.md、不知道系统内部结构。如果审计员在 Harness 内部，它会不可避免地接触到代码上下文，失去"陌生用户"视角。
2. **身份纯粹** — 审计员的唯一身份是"你系统的用户"，不是"你团队的 QA"。它的困惑和失败代表的是真实用户（包括 AI Agent）的体验。
3. **不可被影响** — Harness 内部的 Evaluator 会读 features.json 的 acceptance 标准来验收，这是"按规格检验"。审计员不知道有什么规格，只会问"这个东西好不好用"。

| 对比 | AI 审计员 | Harness Evaluator |
|------|---------|------------------|
| 位置 | 工程流程外部 | 工程流程内部 |
| 知识 | 零知识，只靠 Tool 探索 | 读 features.json、acceptance 标准 |
| 视角 | "这个系统好不好用" | "这个功能是否符合验收标准" |
| 产出 | 审计报告（问题 + 评分 + 建议） | PASS/FAIL 判定 |
| 触发 | 每轮迭代完成后，人工安排 | 状态机自动触发（verifying 阶段） |

两者互补：审计员发现"应该做什么"，Harness 确保"做对了"。

### 5.2 Harness 工程体系

审计报告中的发现，通过 Harness 状态机转化为代码修复：

```
new → planning → building → verifying → fixing ⟷ reverifying → done
```

| 角色 | 职责 | 核心原则 |
|------|------|---------|
| Planner | 分析审计报告、拆解功能 | 不猜需求，所有改进来自审计发现 |
| Generator | 逐条实现 | 不自评，代码由 Evaluator 独立验收 |
| Evaluator | 按 acceptance 标准验收 | 不修代码，只记录问题和证据 |

### 5.3 反馈闭环追踪

每条审计发现从报告到解决全程可追溯：

| 审计报告 | 发现 | 解决批次 | 状态 |

## 六、关键教训

### 安全修复必须全路径排查

generate_image 的脱敏经历了三轮：chat 路径 → routing catch → adapter catch + REST API。**发现一处安全问题后，grep 所有相似路径。"修了报告提到的那个"不等于"修完了"。**

### 双重独立验证：外部审计 + 内部验收

本项目有两层独立验证：外部 AI 审计员（发现"应该做什么"）和内部 Evaluator（确保"做对了"）。error_code 功能经历 5 轮内部修复，是 Evaluator 在每轮都发现新的未覆盖路径。而 generate_image 脱敏遗漏则是外部混沌测试复测才发现的——内部 Evaluator 按 acceptance 验收已通过，但审计员从用户视角发现仍有泄露。**两层独立验证缺一不可。**

### 设计稿是持续约束而非一次性交付

1:1 还原完成 6 小时后被功能批次重写 300 行代码。**功能改造批次的 acceptance 必须包含"变更后页面布局与设计稿一致"。**

### 提示词是可复用的测试资产

同一份提示词多轮执行，清晰展示修复进度。**提示词不要改，改了就失去了可比性。**

### 并行审计要隔离账号

FinOps 审计员看到混沌测试员的调用，误判为"未授权 Prompt Injection 攻击"。**不同角色的审计应使用独立的项目/API Key。**

## 七、复用指南

### 7.1 最小启动（30 分钟）

1. 准备 6 份提示词（参考 `docs/case-studies/prompt/`）
2. 创建独立测试账号
3. 用 3 个独立的 AI 会话分别执行 DX / 混沌 / FinOps 审计
4. 收集报告，分析定级，进入修复循环

### 7.2 每轮迭代检查清单

- [ ] 执行 3+ 个角色的审计
- [ ] 保存报告到 `docs/test-reports/user_report/`
- [ ] 逐条分析发现，验证是否仍存在
- [ ] 按 P0/P1/P2 定级，写入 backlog
- [ ] 批次实现 + 独立验收
- [ ] 部署后用**同一提示词**复测

### 7.3 核心原则

- **不猜需求** — 所有改进来自 AI 审计发现，不是内部臆测
- **不自评** — 实现和验收必须是不同的 agent
- **不改提示词** — 同一提示词跨轮次对比才有意义
- **不只修一个** — 发现一处问题，排查所有相似路径
- **踩过的坑变铁律** — 教训写入框架约束，防止再犯

#### 提示词 1：DX 逆向工程审查

**角色：** 零知识 DX 审查员
**目标：** 在没有任何文档的情况下，纯靠 Tool 探索理解系统全貌并评分
**使用场景：** 每轮迭代必做，是最核心的审计

```markdown
你现在是一位极其挑剔的资深架构师和开发者体验（DX）审查员。你刚刚获得了一个未知系统提供的 MCP (Model Context Protocol) 端点接入权限。你没有任何关于这个系统的外部文档、白皮书或源代码。

你的任务是：**纯靠探索 MCP 提供的 Tools 和返回值，逆向推导出这个系统的完整全貌，并给它的开发者体验打分。**

**请严格按照以下 3 个步骤执行并输出报告：**

## 第一步：全量资产盘点 (Tool Discovery & Mapping)
请自主调用必要的内部指令或工具，遍历当前 MCP Server 暴露的所有 Tools。
- 列出你发现的所有 Tool 名称。
- 根据这些 Tool 的入参（Schema）和描述（Description），用你自己的话解释它们分别是用来干什么的。

## 第二步：系统能力逆向推演 (Reverse Engineering)
基于你发现的工具链，告诉我：
- 这是一个什么性质的平台？它的核心商业逻辑是什么？
- 对于一个新接入的开发者来说，这个平台的核心使用流程（Workflow）应该是怎样的？

## 第三步：极客视角的吐槽与建议 (DX Critique & Suggestions)
现在，以最严格的标准，审查这些 MCP 工具的设计质量。请直接给出至少 3 条切实可行的优化建议（不仅限于以下方面）：
- 某些 Tool 的描述是否过于简略，存在歧义？
- 参数校验（Schema validation）是否足够严谨？有没有缺少必要的枚举值（Enum）或默认值提示？
- 错误信息（如果有）是否具备自解释性？有没有缺少关键的排错引导？
```

#### 提示词 2：混沌工程与边界测试

**角色：** 安全工程师 / 混沌工程师

**使用场景：** 每轮迭代必做，重点关注安全和稳定性


#### 提示词 3：FinOps 一致性审计

**角色：** 财务审计员
**目标：** 定量验证计费精度，三方闭环对账
**使用场景：** 涉及计费/扣费改动后必做

